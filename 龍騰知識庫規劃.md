將 Nanobot 改寫為高中學科解題 RESTful API

我已經完整讀過 nanobot 的架構，以下是我的分析：

---
可行性：非常適合

Nanobot 的架構天然適合這個改造，原因：

1. 已有解耦的 MessageBus — Channel 與 AgentLoop 之間透過 InboundMessage / OutboundMessage 通訊，加一個 HTTP Channel 就能接入 REST API
2. Tool 系統可擴展 — 加入「知識庫檢索」工具只需繼承 Tool base class
3. Session 管理已存在 — 每個學生可以有獨立的 session key

關鍵設計決策需要討論

1. 知識檢索策略：Agent 自主判斷 vs 強制檢索？

建議讓 Agent 自主判斷。做法是註冊一個 KnowledgeRetrieveTool，在 system prompt 中告訴 Agent：

▎ 「你是高中學科解題助教。當學生提問涉及特定知識點時，先用 knowledge_search
工具查詢知識庫，再結合檢索結果回答。如果是簡單問題（如計算題），可以直接回答。」

Agent 會透過 tool_calls 機制自行決定是否呼叫這個工具，這正是 nanobot 的 agent loop 已經支援的模式。

2. 知識上傳與自動歸類

這是最需要設計的部分。建議分兩階段：

- 解析階段：上傳檔案 → 用 LLM 提取知識點 → 拆成小段落（chunk）
- 歸類階段：LLM 判斷每個 chunk 的學科/章節 → 存入對應資料夾

可以直接用 nanobot 的 AgentLoop 本身來做這件事——上傳時觸發一個特殊的 inbound message，讓 Agent 用 filesystem tools 把知識存到正確位置。

3. 檢索方式：向量搜尋 vs 關鍵字？

┌─────────────────────┬──────────────┬────────────┐
│        方案         │     優點     │    缺點    │
├─────────────────────┼──────────────┼────────────┤
│ 純檔案 + 關鍵字     │ 簡單、零依賴 │ 語義匹配弱 │
├─────────────────────┼──────────────┼────────────┤
│ 嵌入向量 + ChromaDB │ 語義搜尋準確 │ 多一個依賴 │
├─────────────────────┼──────────────┼────────────┤
│ 混合（推薦）        │ 兩者兼得     │ 稍微複雜   │
└─────────────────────┴──────────────┴────────────┘

考慮到是高中學科，知識量不會太大（幾百到幾千個文件），先用純檔案結構 + LLM 關鍵字搜尋就夠了，之後有需要再加向量搜尋。
