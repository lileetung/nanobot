⏺ nanobot 處理了三種不同的「思考」機制，因為各家 API 回傳方式不同：                                     
                    
  1. content 裡夾帶 <think> 標籤 — DeepSeek-R1 等                                                       
                                                                                                        
  有些模型把思考過程直接塞在回覆文字裡：                                                                
                                                                                                        
  <think>用戶問天氣，我應該呼叫搜尋工具...</think>                                                    
  我來幫你查一下台北天氣。

  nanobot 用 _strip_think() 把它剝掉，只顯示乾淨的內容：

  # L162-167
  def _strip_think(text):
      return re.sub(r"<think>[\s\S]*?</think>", "", text).strip()

  2. reasoning_content — Kimi、DeepSeek-R1 的 API 欄位

  這些模型的 API 回傳裡有獨立的 reasoning_content 欄位，跟 content 分開：

  {
    "content": "台北現在 18°C",
    "reasoning_content": "用戶問天氣，我需要搜尋..."
  }

  3. thinking_blocks — Anthropic Claude extended thinking

  Claude 用結構化的 thinking blocks：

  {
    "content": "台北現在 18°C",
    "thinking": [
      {"type": "thinking", "thinking": "用戶問天氣..."}
    ]
  }

  在 _run_agent_loop 裡怎麼處理

  # L203-208 — 有 tool_calls 時，先把思考內容發給用戶看
  if response.has_tool_calls:
      thought = self._strip_think(response.content)   # 剝掉 <think> 標籤
      if thought:
          await on_progress(thought)                   # 顯示中間思考
      await on_progress(self._tool_hint(...))          # 顯示「正在使用 web_search(...)」

  # L221-225 — 三種思考都存進歷史，讓 LLM 下一輪能看到
  messages = self.context.add_assistant_message(
      messages, response.content, tool_call_dicts,
      reasoning_content=response.reasoning_content,    # Kimi/DeepSeek
      thinking_blocks=response.thinking_blocks,        # Claude
  )

  各家 API 的思考能力比較

  ┌─────────────────────┬────────────────────────┬───────────────────────────┬─────────────────────┐
  │        模型         │     提供思考過程？     │         回傳方式          │      對應欄位       │
  ├─────────────────────┼────────────────────────┼───────────────────────────┼─────────────────────┤
  │ DeepSeek-R1         │ 有                     │ <think> 在 content 裡 +   │ reasoning_content   │
  │                     │                        │ 獨立欄位                  │                     │
  ├─────────────────────┼────────────────────────┼───────────────────────────┼─────────────────────┤
  │ Kimi (Moonshot)     │ 有                     │ 獨立欄位                  │ reasoning_content   │
  ├─────────────────────┼────────────────────────┼───────────────────────────┼─────────────────────┤
  │ Claude (extended    │ 有                     │ 結構化 block              │ thinking_blocks     │
  │ thinking)           │                        │                           │                     │
  ├─────────────────────┼────────────────────────┼───────────────────────────┼─────────────────────┤
  │ OpenAI o1/o3        │ 有思考但不回傳完整內容 │ 只給摘要                  │ nanobot             │
  │                     │                        │ reasoning_summary         │ 目前沒特別處理      │
  ├─────────────────────┼────────────────────────┼───────────────────────────┼─────────────────────┤
  │ GPT-4o / Gemini     │ 沒有                   │ 不提供思考過程            │ 無                  │
  └─────────────────────┴────────────────────────┴───────────────────────────┴─────────────────────┘

  所以回答你的問題：
  - OpenAI o1/o3 有內部思考，但 API 只給摘要，不給完整推理鏈
  - GPT-4o 沒有思考過程
  - Gemini 的 Thinking 模式有，但 nanobot 目前沒有針對它做特別處理
  - 真正能拿到完整思考內容的是 DeepSeek-R1、Kimi、Claude

  nanobot 的設計很務實——用 LLMResponse 統一三種格式，_run_agent_loop
  不需要知道底層是哪家模型，只管檢查這三個欄位有沒有值就好。
